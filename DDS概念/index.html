DDS概念

Created2025-10-19|Updated2025-10-19
|Post Views:

概要

DDS 中用于区分实体与通信范围的主要唯一标识有：DomainId、Participant GUID、Topic 名称与类型、DataWriter/DataReader（或 RTPS writer/reader）GUID、以及 Topic 实例的 key 字段等.

主要标识项

  • DomainId

    • 一个无符号整数（通常 32 位）用来划分独立的通信域，只有相同 DomainId 的 Participant 才会发现彼此与通信.

  • DomainParticipant 标识（GUID）

    • 每个 DomainParticipant 在底层有唯一的标识（通常为 GUID/ParticipantId），由实现（RTPS 层）生成并用于发现与路由.

  • Topic 标识（name + type）

    • Topic 的唯一性由 TopicName 决定，通常还需要配合 typeName（数据类型名）来唯一识别一个主题；不同 type 或不同 name 的 Topic 被视为不同实体.

  • DataWriter / DataReader（RTPS Writer/Reader）GUID

    • 发布者和订阅者在 RTPS 层有独立的 GUID，用于对等发现、流控与数据路由，多个 Writer 可以发布同一 Topic，但每个 Writer 由唯一 GUID 区分.

  • Topic 实例 key（Data Instance Key）

    • 对于有 key 的类型，某条数据实例由 key 字段的值唯一标识，DDS 在 history、生命周期和读取语义上使用实例 key 做区分（即同一 Topic 下不同实例）.

其它相关可用标识与属性

  • Participant 的 QoS、租约/lease、participant name（可读性字段）等虽然不是唯一标识，但常用于选择、筛查与策略匹配.
  • Discovery 中还存在 locator（IP:port）与通道级别标识，用于路由与传输定位；这些与 GUID 结合实现端到端寻址.

使用建议（工程视角）

  • 当需要全局隔离或多租户，使用不同 DomainId；当需要同域中更细粒度控制，结合 Participant GUID、partition 或 content-filtered topic 来实现.
  • 在程序层面将 Topic 唯一性视为 name+type，若需要区分实例数据则设计并使用 message key 字段.

场景概述

把 DDS 的标识比喻成两辆汽车的仿真测试现场：两辆车 A 和 B 在同一测试场（或不同场），每辆车上有多个子系统（传感器、控制器、记录器），它们通过不同的“话题”互相交换信息。下面把 DDS 的主要唯一标识逐一对应到这个场景，给出直观示例与工程含义。

DomainId —— 测试场或车队隔离

  • 比喻：不同的测试场地或隔离的车队。DomainId 相同的参与者才能发现彼此并通信。
  • 示例：DomainId=1 表示“城市路况测试场”；DomainId=2 表示“高速场景”。把车 A 和车 B 放在 DomainId=1，就能互相看到对方的数据；若车 B 在 DomainId=2，则看不到 A 的消息。

DomainParticipant（Participant GUID）—— 单辆车上的网络身份（或进程身份）

  • 比喻：每辆车的车载网络节点（或车内某个进程/ECU）的唯一车牌/身份标识。
  • 示例：车 A 上有“定位 ECU（Participant A_loc）”、“控制器（Participant A_ctrl）”、“记录器（Participant A_rec）”。每个 Participant 底层都有唯一 GUID，用于发现与路由。

Topic 名称 + Type —— 车内/车间的数据通道（话题和数据格式）

  • 比喻：通信频道的名称和数据格式，例如“里程表”“GPS位姿”“雷达点云”。
  • 示例：Topic name = “vehicle/odometry”，type = “Odometry”。车 A 的定位 ECU 和车 B 的控制器要交换位姿，就要订阅相同的 topic name 和兼容的 type。

DataWriter / DataReader（Writer/Reader GUID）—— 具体发布者或订阅器实例

  • 比喻：同一话题上不同设备或不同实例的“发话人”和“听话人”。即使话题相同，不同的设备/实例仍有各自身份。
  • 示例：车 A 的前置雷达有一个 DataWriter（Writer GUID），后置雷达可能有另一个 Writer；同一控制台上可以有多个 Reader，每个 Reader 都有唯一标识，用于流量控制与点到点识别。

Topic 实例 Key（数据实例 key）—— 相同话题下的具体对象（车辆 ID / 传感器 ID）

  • 比喻：话题下的“对象身份”，比如“车辆ID=CarA”或“轮胎ID=rear_left”。Key 将同一 topic 中不同实体的数据区分开。
  • 示例：Topic “vehicle/pose” 有 key 字段 vehicle_id；来自 CarA 的实例 key=”CarA”，来自 CarB 的 key=”CarB”。Reader 可按实例选择性处理或保留历史（history per instance）。

Locator (IP:port) 与网络层标识 —— 物理连接位置

  • 比喻：车辆连接到哪根网线或哪个无线接口（具体到车内以太网端口或主机地址）。用于路由到具体的传输端点。
  • 示例：Participant A 的 locator = 192.168.1.10:7400，Participant B 在 192.168.1.11:7400，用于具体数据包发送/接收定位。

辅助 / 策略性标识（partition、participant name、QoS）

  • Partition：比喻车内的逻辑隔间（如 infotainment 区、safety 区），即便在同 Domain 内也能将流量逻辑隔离。
  • Participant name：可读标签（如 “CarA_General”），便于人工识别但不是底层唯一 GUID（可用于筛查与日志）。
  • QoS：比喻话题的通信约定（可靠/不可靠、历史深度、延迟容忍度），决定消息传输的行为和性能。

具体应用举例（两车仿真）

  • 场景 1（同域交互）：两车都在 DomainId=1；它们各自的定位 Participant 发布 topic “vehicle/pose”；控制器订阅该 topic，并通过 key=vehicle_id 区分 CarA/CarB 的位姿。这样可以在同一仿真场内直接看到对方位置。
  • 场景 2（隔离测试）：把车 B 放在 DomainId=2，或在同 Domain 但放到 partition=”internal”；则车 A 无法接收到 B 的数据，实现网络级或逻辑级隔离。
  • 场景 3（多实例同 topic）：车 A 上两个相同型号的雷达分别作为两个 DataWriter 发布 “sensor/pointcloud”；下游汇聚节点通过 writer GUID 或 source locator 区分并合并数据流。

工程建议（基于比喻）

  • 需要全局隔离时改 DomainId；需要同域但逻辑隔离用 partition；需要数据级区分用 message key（vehicle_id）；需要识别具体发布源则依赖 Writer/Participant GUID 或 locator。
  • 设计 topic 和 key 时优先考虑可扩展性（例如对 fleet 仿真，使用 vehicle_id 作为 key），并配合合适的 QoS（实时控制用低延迟/可靠性设置），以满足仿真精度与可靠性需求。

映射原则（物理世界到软件实体）

  • 把每个 DDS 实体映射为一个物理角色或物理对象的“责任域”。
  • 以“谁拥有/发布/感知这个通道”为核心来命名 channel 的归属。
  • 保持两套标识：友好语义名 用于配置与阅读，底层标识（GUID、DomainId）用于调试和互操作。

建议的物理映射与命名模式

  • DomainId → workspace_id
    • 表示一个物理测试场地、网络隔离域或车队。
  • DomainParticipant → owner 或 host 或 unit
    • 表示“谁”在该 workspace 内拥有设备或进程。优先用 owner 表达“归属关系”。
  • Topic → channel
    • 表示数据流的“通道”。channel 名应以“所属/用途”前缀表示归属。
  • DataWriter → channel_sender 或 sender
    • 表示发送该 channel 数据的具体设备或进程。
  • DataReader → channel_receiver 或 receiver
    • 表示接收该 channel 数据的具体设备或进程。
  • Topic 实例 key → instance_id 或 object_id
    • 表示同一 channel 下不同物理对象的标识（例如 vehicle_id）。
  • Locator / endpoint → endpoint 或 transport_address
    • 表示物理网络位置或接口。

channel 的归属（如何表达“谁的 channel”）

  • 命名约定： “/“
    • 例： CarA/pose 表示属于 CarA 的位置通道。
  • 当 channel 为共享资源（fleet 层）时使用 tenant 或 group 前缀： Fleet/traffic/pose。
  • 若 channel 代表虚拟聚合或中间件服务，使用 service 前缀： Infra/time_sync、Logger/event_stream。

具体两车仿真命名示例

  • workspace_id = 1
  • owner = CarA; owner = CarB
  • channel = CarA/pose; channel = CarB/pose; channel = Fleet/obstacles
  • channel_sender: CarA_gps_sender; CarA_radar_sender
  • channel_receiver: CarB_pose_receiver; Logger_event_receiver
  • instance_id: vehicle_id=CarA; sensor_id=front_radar_01
  • endpoint: CarA:192.168.10.11:7400

日志与配置建议（便于排查）

  • 在日志和管理界面同时记录友好名与底层 id：
    • 格式示例： workspace_id=1 owner=CarA owner_guid=… channel=CarA/pose channel_type=Odometry sender_id=CarA_gps_sender writer_guid=…
  • 在配置文件中使用明确字段名：
    • workspace_id, owner.name, owner.guid, channel.name, channel.type, channel.instance_id, endpoint.address

最终推荐简短命名表

  • DomainId -> workspace_id
  • DomainParticipant -> owner 或 unit（推荐 owner）
  • Topic -> channel
  • DataWriter -> channel_sender 或 sender
  • DataReader -> channel_receiver 或 receiver
  • Topic instance key -> instance_id 或 object_id
  • Locator -> endpoint

根本概念与核心实体

  • DomainId
    • 通信域边界，决定能否相互发现与通信。类似不同测试场地或车队。
  • DomainParticipant
    • 域内的进程或实体代理，负责注册本进程内的发布/订阅并参与发现。每个 Participant 有底层 GUID 和可读 name。
  • Topic
    • 主题名和类型联合定义的数据通道，声明消息格式与语义。Topic 本身不发送数据，只描述通道。
  • Publisher 和 DataWriter
    • Publisher 是发布端的集合/容器，DataWriter 是具体的发送者实例。Writer 对应底层 writer GUID 和传输 endpoint。
  • Subscriber 和 DataReader
    • Subscriber 是订阅端的集合/容器，DataReader 是具体的接收者实例。Reader 对应 reader GUID。
  • Instance Key
    • 如果消息类型定义了 key 字段，同一 Topic 下不同 key 代表不同数据实例（历史、生命周期按实例区分）。
  • Locator 和 QoS 等元信息
    • Locator 表示网络地址，QoS/partition 等为策略与逻辑隔离参数，用于控制可见性、可靠性、历史等。

父子关系与基数（谁属于谁 一对一 一对多）

  • DomainId 包含 0..N DomainParticipant。
    • 基数 1..N：一个 workspace 可以有多参与者。
  • DomainParticipant 包含 0..N Publisher 和 0..N Subscriber，以及可创建的 Topic (引用全域 Topic)。
    • Participant 是 publisher/subscriber 的宿主（ownership）。
  • Publisher 包含 0..N DataWriter；Subscriber 包含 0..N DataReader。
    • 一对多：一个 Publisher 管理多个 Writer；一个 Subscriber 管理多个 Reader。
  • DataWriter 和 DataReader 指向同一个 Topic（多对多关系）：多个 Writers 可写同一 Topic，多 Readers 可读同一 Topic。
  • Topic 下可存在 0..N instance_id（若定义 key）
    • 一个 Topic 对应多实例；读写/历史按 instance key 区分。
  • Writer/Reader GUID 与 endpoint 一一对应（运行时唯一标识）。

简化链路视图：

  • DomainId -> DomainParticipant -> {Publisher -> DataWriter, Subscriber -> DataReader} -> Topic -> instance_id

各实体职责（聚焦行为与边界）

  • DomainId
    • 职责：划分发现和路由边界，隔离租户或测试场地。
  • DomainParticipant
    • 职责：在域内注册本进程的 Topics、Publisher、Subscriber；参与 discovery；持有 participant 级 QoS（lease、liveliness）。
  • Topic
    • 职责：定义消息格式、name/type 唯一性、是否有 key；不直接传输数据。
  • Publisher
    • 职责：管理一组 Writer，共享某些 QoS（可选）；负责数据发送策略的上层封装。
  • DataWriter
    • 职责：把样本序列化并发送到网络；维护 writer GUID、history、deadline、lifespan 等 writer 级 QoS。
  • Subscriber
    • 职责：管理一组 Reader，接收和分派数据到应用回调或队列。
  • DataReader
    • 职责：接收样本、维护实例状态、按 QoS 交付给应用（读取、监听），支持按 instance key 过滤/历史。
  • Locator / Endpoint
    • 职责：物理地址定位与路由，RTPS 层用于建立点到点传输路径。
  • QoS 与 Partition
    • 职责：控制可见性、可靠性、保留历史、延迟容忍等行为；partition 提供逻辑隔离。

Publisher Subscriber 与 Writer Reader 的对应与差异

  • 对应关系
    • Publisher 包含 DataWriter；Subscriber 包含 DataReader。
    • DataWriter 对应发送端具体实体，DataReader 对应接收端具体实体。
  • 角色差异与使用场景
    • Publisher/Subscriber 是更上层的管理抽象，便于批量管理 QoS 与生命周期。
    • Writer/Reader 是实际发送/接收单元，控制序列号、instance 历史、心跳、流控。
  • 多对多模型与路由
    • 多个 Writers -> 同一 Topic -> 多个 Readers。Reader 按 QoS 决定是否接收、按 instance key 组织历史。
  • discovery 与匹配
    • Participant 发现彼此后，Writer/Reader 在 RTPS 层通过匹配对方的 Topic name/type 与 QoS 来建立数据路径。

物理世界比喻结合代码理解（两车仿真快速对应）

  • workspace_id = 车场（DomainId）
  • owner = 每辆车上的进程/ECU（DomainParticipant）
  • channel = 某条数据流（Topic name+type），例如 CarA/pose
  • channel_sender = GPS 进程的 DataWriter；channel_receiver = 控制器的 DataReader
  • 多个 GPS（前后）就是多个 DataWriter 发布同一 channel；多个控制器或记录器就是多个 DataReader 订阅同一 channel
  • instance_id = vehicle_id 字段把 CarA/CarB 在同一 channel 下区分开
  • 在代码层面（伪结构）：
    • participant = DomainParticipant(domainId)
    • topic = participant.create_topic(“CarA/pose”, Odometry)
    • pub = participant.create_publisher(qos)
    • writer = pub.create_datawriter(topic, writer_qos)
    • sub = participant.create_subscriber(qos)
    • reader = sub.create_datareader(topic, reader_qos)

实践要点（帮助理解架构）

  • 想清楚“所有权”時首看 DomainParticipant：它是进程/车上实体的根，负责注册和持有 publishers/subscribers。
  • 想清楚“数据流”時看 Topic：Topic 定义消息语义，Writer/Reader 是运行时的实际发/收端。
  • 观察匹配与隔离：DomainId 与 partition 决定能否看见对方；QoS 决定是否能通信（可靠性、历史、延迟等）。
  • 日志设计：在每条日志同时记录 friendly name（owner/channel/sender）与底层 id（participant GUID / writer GUID / domainId / endpoint）以便排查。
